# 决策记录

## ADR-0001：采用单体优先的架构策略
- 状态：Accepted
- 决策：当前阶段保持单体架构
- 原因：优先形成完整考试闭环，降低复杂度

## ADR-0002：主链路优先于高级扩展
- 状态：Accepted
- 决策：优先实现题库、试卷、考试、作答、阅卷、成绩主链路
- 原因：避免过早被 AI、复杂监考、编程题等高复杂度能力拖慢

## ADR-0003：使用仓库文档作为长期项目记忆
- 状态：Accepted
- 决策：将计划、记录、决策、交接、评测写入仓库
- 原因：保证长期自治开发可追溯、可交接

## ADR-0004：防作弊采用基础实现 + 高阶预留
- 状态：Accepted
- 决策：先实现浏览器侧异常检测、事件落库、后台查询，再逐步增强
- 原因：当前阶段优先保证核心考试链路稳定，同时保留扩展空间

## ADR-0005：试卷题目组合采用关系表实现
- 状态：Accepted
- 决策：使用 `biz_paper_question` 存放试卷题目关系
- 原因：关系表更利于审计、分析、阅卷与后续扩展

## ADR-0006：保留 H2 作为快速启动验证模式
- 状态：Accepted
- 决策：正式数据库为 MySQL，H2 仅用于本地快速验证
- 原因：兼顾交付口径与开发效率

## ADR-0007：菜单可见性由后端控制
- 状态：Accepted
- 决策：根据角色在后端过滤菜单
- 原因：安全边界应尽量收敛在服务端

## ADR-0008：考试时间语义拆分为“进入窗口”与“作答时长”
- 状态：Accepted
- 决策：
  - `startTime / endTime` 用于表达学生允许进入考试的时间窗口
  - 学生实际作答截止时间按 `min(startedAt + durationMinutes, endTime)` 计算
- 原因：避免把“还能不能进入考试”和“还能答多久”混为一谈

## ADR-0009：试卷蓝图采用“关系表 + JSON 配置”混合存储
- 状态：Accepted
- 决策：
  - 题目明细继续存于 `biz_paper_question`
  - 题型分布、难度分布等蓝图配置存于 `biz_exam_paper` JSON 字段
- 原因：既保留精确卷面明细，又便于保存组卷策略与后续扩展

## ADR-0010：学生端乱序采用稳定随机而不是每次重新打乱
- 状态：Accepted
- 决策：当试卷开启乱序时，按答卷维度稳定计算题目和选项展示顺序
- 原因：既满足防作弊需求，又避免学生刷新页面后题序变化导致体验混乱

## ADR-0011：前端路由必须与菜单和接口权限保持一致
- 状态：Accepted
- 决策：
  - 前端在进入受保护页面前，必须根据当前账号的可见页面菜单做路由拦截
  - 对隐藏但可访问的业务子路由（如试卷创建、试卷编辑），按其所属页面前缀做兼容判断
- 原因：
  - 本轮核查确认学生可直接访问 `/exam/proctor`，页面骨架能加载但接口返回 403，形成明显半残状态
  - 权限一致性直接影响验收观感，因此需要将“菜单权限、路由权限、接口权限一致”明确为工程决策

## ADR-0012：错题本基础版采用“已发布成绩动态聚合”策略
- 状态：Accepted
- 决策：
  - 错题本基础版不单独新增错题沉淀表，先按已发布成绩动态聚合 `biz_score_record`、`biz_answer_item`、`biz_question_bank`
  - 学生考试中的“待复查”状态落到 `biz_answer_item.review_later_flag`
- 原因：
  - 当前仓库已经具备成绩、答题项、题目元数据三类完整数据，动态聚合能更快形成真实可用能力
  - “错题沉淀”与“考试中待复查”语义不同，前者可动态生成，后者需要真实持久化

## ADR-0013：基础监考策略采用“配置中心下发 + 前端拦截 + 后端留痕”模式
- 状态：Accepted
- 决策：
  - 使用现有 `sys_config_item` 承载 `anti_cheat` 分组配置
  - 考试工作区按考试的 `antiCheatLevel` 和全局配置组装下发监考策略
  - 学生端负责复制/粘贴/右键/高风险快捷键拦截
  - 后端统一记录事件、IP、设备指纹与设备摘要
- 原因：
  - 该方案改动范围可控，适合当前单体架构快速形成真实闭环
  - 不需要先建设独立的监考策略中心页面，也能满足基础版验收

## ADR-0014：登录风险记录与单设备限制先采用“基础日志 + 考试态约束”模式
- 状态：Accepted
- 决策：
  - 登录阶段先记录成功/失败日志、IP、设备指纹和基础风险原因
  - 单设备限制先只对严格考试态下的考试工作区进入生效
  - 登录态全局会话治理、设备踢出和并发会话管理后置
- 原因：
  - 这种实现更轻，更适合作为当前阶段的验收加分项
  - 能与现有监考事件和设备指纹能力直接复用，快速形成闭环

## ADR-0015：认证安全策略采用“用户表状态 + 配置中心阈值 + 站内告警”模式
- 状态：Accepted
- 决策：
  - 账号失败次数、最近失败时间、锁定截止时间落到 `sys_user`
  - 认证安全阈值统一落到 `sys_config_item` 的 `auth_security` 分组
  - 触发账号锁定或 IP 限流时，先通过现有站内消息通道发送 `SECURITY_ALERT`
- 原因：
  - 不引入额外中间件即可形成“阈值可调、状态可查、告警可达”的基础安全闭环
  - 复用现有配置中心和消息中心，可以在当前单体架构下快速落地并保持可维护性

## ADR-0016：通知协同采用“模板驱动 + 统一分发 + Mock 外发通道”模式
- 状态：Accepted
- 决策：
  - 新增 `sys_notification_template` 统一维护通知模板
  - 新增 `NotificationService` 统一承接考试发布、开考前提醒、成绩发布、申诉结果和安全告警
  - 当前外发通道先落 `MOCK_SMS`，并通过 `biz_notification_delivery_log` 留痕
  - 开考前提醒同时提供定时扫描与手动触发入口
- 原因：
  - 直接在各业务服务里拼装站内消息会导致文案分散、日志缺失且难以扩展真实通道
  - 当前阶段先用 Mock 通道打通“模板 -> 渲染 -> 投递 -> 留痕”闭环，既满足验收，也为后续接短信/邮件网关保留边界

## ADR-0017：列表页时间展示统一走本地化格式化工具
- 状态：Accepted
- 决策：
  - 前端新增统一时间格式化工具 `frontend/src/utils/datetime.ts`
  - 所有面向交付的列表页、详情页和消息页统一使用本地化时间展示
- 原因：
  - 直接展示 ISO 时间串会明显拉低交付观感
  - 时间口径统一后，更利于截图、验收和后续维护

## ADR-0018：考试期保护聚焦“高风险更新 / 删除”，不阻断新内容准备
- 状态：Accepted
- 决策：
  - 对进行中考试已关联内容继续执行高风险更新 / 删除保护
  - 不再因存在进行中考试而阻断教师准备新的题目和新的试卷
- 原因：
  - 全局禁止新增会把教师备课与下场考试准备动作一并锁死，破坏主链路可用性
  - 真正需要保护的是已关联进行中考试的存量内容，而不是与当前考试无关的新内容
  - 用户可见的时间字段统一输出为本地化日期时间，而不是直接展示后端原始字符串
  - 考试发布、学生待考、通知模板、通知投递日志、消息中心、公告、审计日志、登录风险和监考事件页统一复用该工具
- 原因：
  - 原始 ISO 或后端字符串直接展示会让页面显得偏开发态，不符合正式交付观感
  - 统一工具可以减少重复代码，降低后续时间格式不一致的风险
  - 该调整只影响展示层，不改变业务规则与数据结构

## ADR-0018：通知模板与公告采用“组织隔离 + 全局兜底”策略
- 状态：Accepted
- 决策：
  - 为 `biz_notice` 与 `sys_notification_template` 补充 `organization_id` 字段
  - 非管理员仅能修改本组织范围内的公告和通知模板
  - 通知模板运行时采用“同组织模板优先、全局模板兜底”的解析顺序
- 原因：
  - 数据隔离要求不能只停留在菜单和页面层，必须落到服务端查询与写操作边界
  - 通知模板存在默认模板复用需求，完全按组织硬拆会增加维护成本；“组织覆盖 + 全局兜底”更适合当前阶段

## ADR-0019：签到规则与准考证基础版直接挂在考试计划上
- 状态：Accepted
- 决策：
  - 在 `biz_exam_plan` 上增加签到开关和提前开放分钟数
  - 在 `biz_exam_candidate` 上记录签到状态与签到时间
  - 学生端通过独立的签到接口完成签到，通过准考证接口查看与打印基础通知单
  - 进入考试工作区前由服务端校验是否已完成签到
- 原因：
  - 这种方案可以最大限度复用现有“考试计划 -> 考生分配 -> 学生待考列表 -> 进入考试”主链路
  - 不需要额外引入复杂签到活动表或生成服务，就能形成可验证、可交付、可继续扩展的基础闭环
