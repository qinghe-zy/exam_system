# Findings

## 本轮新增发现
1. 监考与防作弊能力在当前架构下最适合走“配置中心下发基础策略 + 考试工作区前端拦截 + 后端事件留痕 + 监考页展示”这条闭环路径。
2. 不必先做独立的监考策略中心页面，先把 `anti_cheat` 分组配置项落到现有配置中心，就能形成基础版可配置能力。
3. IP 信息可以直接由服务端从请求上下文识别；设备信息和设备指纹则适合由前端生成后随事件上报。
4. 既有教师策略组卷回归用例对“保存后跳回列表”的假设偏脆，已收口为更稳定的 API 轮询断言。

## 当前仍未完成的核心问题
1. 摄像头/麦克风等硬件依赖能力仍未实现。
2. 多老师协同阅卷、仲裁、回评仍未实现。
3. 更深层的组织对比细分、学习画像、智能推荐仍未实现。
4. 更强考试态控制仍未覆盖考试期配置只读保护之外的更多后台模块。

## 2026-04-08 安全与运维增强阶段新增发现
1. 当前登录能力已具备“风险留痕”数据面，最小闭环补法不是新增复杂表，而是在 `biz_login_risk_log` 基础上增加失败窗口判断、锁定时长和 IP 频率限制。
2. `sys_config_item` 已经是现成的参数承载层，适合继续承载认证安全阈值，避免把安全策略写死在代码中。
3. 仓库已有站内消息通道，可直接作为高风险登录告警的可运行 mock 通道，用于补齐“异常告警占位”而不引入外部依赖。
4. 运维侧当前最缺的不是说明性文字，而是可直接执行的脚本与 runbook，因此本轮需要同时补脚本、操作步骤和验证方法。
5. 真实库验证表明：仅更新仓库 SQL 还不够，必须把 `exam_system` 实库同步打补丁，否则备份恢复结果会落后于仓库基线。
6. 基础压测更适合先用登录接口建立采样基线，再逐步扩展到考试保存、交卷和分析等重链路。

## 2026-04-08 阅卷治理阶段新增发现
1. 当前阅卷链路的核心缺口不是“不会打分”，而是“打分后直接发布”，中间缺少复核、重判和申诉处理层。
2. 现有 `biz_grading_record` 已适合承载多轮评分历史，只需增加轮次/动作语义字段，不必推倒重建。
3. 学生侧已有成绩详情和答卷回看入口，最小可行申诉入口适合直接挂在成绩详情页，而不是单独起一个大模块。
4. 管理端最适合先在现有阅卷中心与成绩中心内扩展治理操作，避免新建过多孤立页面导致维护成本上升。
5. 本地与浏览器自动化都表明：IP 限流更适合只统计失败登录，否则会误伤高并发的正常登录回归。

## 2026-04-08 质量报告阶段新增发现
1. 当前分析能力已经具备“指标数据面”，考试质量报告的关键不是再加表，而是把已有指标转成甲方可读的结论、风险和建议。
2. 质量报告最适合复用分析页，而不是另起独立模块，这样更容易保持数据与分析看板一致。
3. Markdown 导出比 PDF 更轻量，也更适合当前阶段快速形成“可交付、可审阅、可追溯”的正式报告基础版。

## 2026-04-08 设备检测阶段新增发现
1. 设备检测基础版最适合走“配置下发 + 前端预检 + 失败阻断 + 事件留痕”这条闭环路径，不需要新增表结构。
2. 当前桌面浏览器默认窗口高度普遍低于 760，因此设备检测阈值需要兼顾真实考试要求和现有桌面回归环境，最终收口到 1200x700 更合理。
3. 设备检测如果只做提示不做阻断，验收价值会明显下降；当前严格考试态下直接阻止进入更符合“正式考试平台”定位。

## 2026-04-08 会话治理阶段新增发现
1. 登录态会话治理最适合先用 `session_version` 这种轻量方案，不需要引入 Redis 或复杂会话中心就能形成“新登录挤下线、登出即失效”的闭环。
2. 浏览器自动化与单账号单会话规则天然冲突，因此 Playwright 全量回归需要改为串行执行，才能保证验证口径和产品规则一致。
3. 考试开始后计划误删误改是高风险问题，直接在服务层阻止比单纯在页面提示更可靠。

## 2026-04-08 广度优先分析阶段新增发现
1. 当前分析域最适合优先补的是“优秀率、组织对比、历次趋势”这类高感知、低耦合能力，不必先做复杂图表或画像系统。
2. 组织对比可以直接复用成绩记录对应用户的组织信息，不需要新增数据库表。
3. 历次趋势先用近几场考试的时间序列即可形成基础版价值，趋势深入部分可以后续再单独做专项文档。
