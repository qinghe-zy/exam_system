# 在线考试系统能力差距分析

## 1. 当前系统总体定位
当前系统不得再按“已完成的一期核心闭环版”对待，而应按“**尚未完成全部需求、必须继续补齐功能的正式在线考试系统项目**”来推进。

这意味着：
- 已有能力不再是停止理由，只能作为继续扩展的基础。
- 所有“基础版实现 / 部分实现 / 仅预留 / 未实现”的能力项都属于当前继续开发范围。
- 本文档从现在开始不仅负责描述差距，还要把差距转成明确的开发任务、阶段目标和验收口径。

## 1.1 当前继续开发总原则
1. 先补会直接影响“正式在线考试系统”定位的能力缺口。
2. 先补会影响主链路、权限一致性、数据正确性和验收观感的缺口。
3. 对复杂度过高或依赖外部资源的能力，允许先做结构完整的基础版或扩展位，但必须明示当前边界。

## 2. 当前已具备的核心能力
### 2.1 已成型的主链路
当前最有价值的部分，是以下主链路已经真实形成闭环：

1. 题库管理
- 教师可进入题库管理页维护题目、答案、解析、知识点、章节、标签和审核状态。
- 当前支持 JSON 导入导出。
- 当前已支持填空题、论述题、材料题基础版，以及富文本 HTML、材料内容和附件 JSON。
- AI 已从“纯预留”提升为“题库辅助基础版”，可生成题目草稿、优化题干与解析。

2. 试卷组装
- 支持手工组卷、随机组卷和基础策略组卷。
- 试卷可保存题序、分值、版本、备注、题型蓝图、难度蓝图和乱序配置。
- 教师可重新打开试卷继续编辑。

3. 考试发布
- 教师可创建考试计划，绑定试卷和考生。
- 可配置开始时间、结束时间、作答时长、考试口令、迟到限制、提前交卷限制、参考次数和自动交卷开关。
- 发布后可自动写入站内消息提醒。

4. 学生作答
- 学生可查看待考列表并进入考试。
- 工作区有答题卡、倒计时、自动保存、手动保存、交卷确认、全屏切换和基础异常行为采集。
- 实际作答截止时间按“进入时间 + 时长”与“考试结束时间”共同计算，逻辑较清晰。

5. 阅卷与成绩
- 客观题可自动判分。
- 主观题可进入阅卷中心人工评分。
- 阅卷完成后会回写成绩记录并触发站内消息提醒。
- 学生已可查看已发布成绩，并通过消息中心进入逐题详情页。

6. 成绩与基础分析
- 已具备平均分、最高分、最低分、及格率、排行榜、分数段、知识点得分率、题目得分率等基础分析。

7. 基础监考与留痕
- 已具备切屏、失焦、退出全屏等行为留痕。
- 监考事件已能关联考试、考生、自动保存和答卷版本。

8. 账号与权限增强基础版
- 当前登录页已补学生注册和找回密码基础版。
- 验证码当前通过邮箱 / 短信 mock 通道完成联调。
- 关键页面已增加按钮级权限基础版。

### 2.2 已能支撑演示/验收/联调的模块
- 登录与 RBAC 基线
- 组织管理、用户管理、考生导入
- 题库管理
- 试卷管理与独立建卷页
- 考试发布
- 学生考试工作区
- 阅卷中心
- 成绩中心与分析页
- 消息中心与公告管理
- 监考事件页
- 配置中心与审计日志

## 3. 当前主要差距

### 3.1 用户与权限侧差距
#### 当前缺口
- 注册、找回密码和验证码已补基础版，但当前仍缺真实邮件/短信通道和更严格的账号治理策略。
- 缺少统一身份认证、OAuth、学校账号接入。
- 数据隔离虽然已有组织范围控制，但仍是基础版。
- 本轮前已存在“学生可直接输入未授权监考路由，页面骨架可见但接口 403”的半残状态，现已修复。

#### 影响范围
- 影响用户生命周期完整度。
- 影响权限体验一致性。
- 影响对学校统一认证的适配能力。

#### 风险级别
- 权限一致性：高
- 统一认证：中
- 注册/找回密码：中

#### 是否影响甲方验收
- 会影响对“权限严谨性”的感知。
- 但在本轮补上前端路由守卫后，核心权限半残问题已被修复。

### 3.2 题库与题型侧差距
#### 当前缺口
- 当前已支持单选、多选、判断、简答、填空、论述、材料题，但仍未覆盖计算题、编程题、听力题、案例题。
- 已支持富文本 HTML 和附件 JSON 基础版，但不支持公式编辑器、音视频题和真正文件上传。
- 缺少查重、收藏、使用统计、错题沉淀。
- 知识点自动组题已补基础版，但尚未与试卷蓝图深度联动。

#### 影响范围
- 限制正式考试的题型覆盖面。
- 限制教研题库治理深度。

#### 风险级别
- 高

#### 是否影响甲方验收
- 会明显影响“平台规格感”和“题库完整度”。

### 3.3 试卷与发布规则侧差距
#### 当前缺口
- 策略组卷仍是基础版，不能按知识点、章节、复杂分值比例做蓝图控制。
- 缺少试卷审核、模板、归档、多套平行卷、不同考场不同试卷等能力。
- 考试发布缺少补考、缓考、批次、签到、准考证等正式考试管理能力。

#### 影响范围
- 影响教师出卷效率和复杂考试组织能力。

#### 风险级别
- 高

#### 是否影响甲方验收
- 对一般演示场景影响可控。
- 对“正式学校考试平台”定位影响明显。

### 3.4 学生考试体验侧差距
#### 当前缺口
- 无设备检测、无附件题、无公式/代码编辑器、无草稿纸、无考后成绩与解析闭环。
- 弱网容错仍偏基础。
- 防刷新、防误触仅靠自动保存，不够彻底。

#### 影响范围
- 影响学生体验完整度。
- 限制复杂考试场景。

#### 风险级别
- 中到高

#### 是否影响甲方验收
- 对演示影响中等。
- 对正式考试体验评分影响较大。

### 3.5 监考与风控侧差距
#### 当前缺口
- 当前监考属于浏览器侧基础留痕，不是完整防作弊系统。
- 无摄像头、麦克风、人脸、活体、IP、设备、异地登录、相似答案、群体异常分析。
- 无复制粘贴限制、右键/快捷键限制。

#### 影响范围
- 影响考试公正性与风控能力。

#### 风险级别
- 高

#### 是否影响甲方验收
- 若甲方强调“正式机考防作弊”，则影响非常明显。
- 若当前只看一期闭环，则可解释为基础版。

### 3.6 阅卷与成绩闭环差距
#### 当前缺口
- 无流水阅卷、回评、仲裁、复核、申诉。
- 无 AI 阅卷建议。
- 学生侧已补“已发布成绩查询 + 逐题详情”，但错题本、个人轨迹、申诉复核仍未实现。
- 无导出成绩单与分析报表。

#### 影响范围
- 影响考试运营治理和教学闭环。

#### 风险级别
- 中到高

#### 是否影响甲方验收
- 对“可用”影响不大。
- 对“成熟度”影响明显。

### 3.7 通知协同与系统运营差距
#### 当前缺口
- 当前只有站内消息和公告。
- 没有短信、邮件、微信/企业微信/钉钉通道。
- 审批流通知和申诉通知未实现。

#### 影响范围
- 影响跨端协同和运营触达。

#### 风险级别
- 中

#### 是否影响甲方验收
- 会影响“平台完整度”印象，但不阻塞核心链路验收。

### 3.8 安全、运维与交付差距
#### 当前缺口
- 无登录风控、限流、防爆破。
- 已补运行时健康检查接口，但仍缺异常告警、性能监控和统一观测面板。
- 无备份恢复脚本和灾备方案。
- 无并发压测与考试高峰保护。

#### 影响范围
- 影响系统上线可靠性。

#### 风险级别
- 高

#### 是否影响甲方验收
- 对纯功能验收影响有限。
- 对“可上线、可长期运行”口径影响很大。

## 4. 下一步优先级建议
### 4.1 必须优先补的
1. 权限一致性继续收口
- 本轮已修复前端路由守卫。
- 下一步应继续做按钮级权限控制和更细粒度数据权限校验。

2. 题库与组卷增强
- 优先补齐题型扩展、富媒体能力、知识点组卷。
- 这是最容易被甲方直接感知的短板。

3. 学生考后闭环
- 增加学生成绩查看、错题回看、个人答卷回放。
- 这会显著提升“完整平台感”。

4. 安全与运维基线
- 至少补健康检查、登录失败风控、备份恢复说明和基础压测。

### 4.2 适合作为第二阶段补的
1. 通知外发通道
2. 更完整的考试规则与批次/考场机制
3. 阅卷复核、申诉、回评
4. 报表导出与质量分析报告

### 4.3 适合作为扩展预留的
1. AI 阅卷建议
2. 摄像头/人脸监考
3. 编程题沙箱
4. 多租户 SaaS
5. 第三方系统深度对接

### 5.1 从产品完整度看
当前系统已经具备真实考试主链路，但这只能说明基础工程底座已成型，不代表功能已补齐。接下来必须按上述阶段任务持续推进，直到差距项被真正实现、补强或明确列为 blocker。

### 5.2 从工程完整度看
代码、接口、MySQL 脚本、Playwright 用例和核心文档之间已经形成较好的对应关系。本轮又进一步修复了权限半残问题，并将 E2E 用例从“写死时间”调整为“相对当前时间”，提高了回归稳定性。

### 5.3 从验收完整度看
当前仓库虽然已经达到“可核查、可运行、可测试、可解释”的状态，但当前验收目标已升级，不再是“一期核心闭环即可”，而是要继续朝《PRD》目标口径补齐功能。因此，后续是否可停，必须以以下条件判断：
- 非已实现能力项是否已经补齐到验收口径
- 无法补齐项是否已经形成真实 blocker 说明
- 代码、数据库、脚本、页面、权限、文档、测试是否一致
