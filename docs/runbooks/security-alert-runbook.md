# 登录安全告警处置 Runbook

## 一、文档目的
本文档用于说明当前系统登录安全告警的触发条件、排查步骤、影响范围、处置动作和后续追踪方式，确保“登录风险留痕”已经升级为“可告警、可处置、可复盘”的基础运维能力。

## 二、当前已接入的告警范围
当前版本已支持以下登录安全事件进入站内告警通道：
1. 账号在锁定窗口内连续失败达到阈值，触发临时锁定。
2. 同一客户端 IP 在限流窗口内登录请求过多，触发 IP 级限流拦截。

当前告警通道：
- 站内消息 `SECURITY_ALERT`

当前默认接收角色：
- `ADMIN`
- `ORG_ADMIN`

配置来源：
- `auth.security.alert.message.enabled`
- `auth.security.alert.recipient.roles`

## 三、排查入口
1. 登录风险记录页：`/system/login-risks`
2. 管理员消息中心：`/messages`
3. 数据表：
   - `biz_login_risk_log`
   - `biz_in_app_message`
4. 关联配置：
   - `auth.security.login.max-failures`
   - `auth.security.login.lock-minutes`
   - `auth.security.login.ip-rate-limit.window-seconds`
   - `auth.security.login.ip-rate-limit.max-attempts`

## 四、标准处置流程
### 4.1 收到“账号触发临时锁定”告警
1. 打开登录风险记录页，按时间倒序查看对应账号。
2. 核对以下字段：
   - 用户名
   - 风险级别
   - IP
   - 设备指纹
   - 风险原因
   - 登录时间
3. 判断是否为以下场景：
   - 用户输错密码
   - 批量撞库/爆破尝试
   - 同账号在多个环境反复登录
4. 处理建议：
   - 若确认是本人误操作：指导用户等待锁定时间结束，或走找回密码流程重置密码
   - 若怀疑被撞库：立即通知账号持有人改密，并持续观察同 IP 的其他账号尝试

### 4.2 收到“登录 IP 触发限流”告警
1. 在登录风险记录页按 IP 搜索最新记录。
2. 判断是否集中攻击：
   - 是否短时间内命中多个用户名
   - 是否来自异常网络出口
3. 处理建议：
   - 若为压测或联调环境，请确认是否为人工脚本触发
   - 若为未知来源，建议临时封禁来源 IP，并保留告警截图与日志

## 五、恢复动作
### 5.1 账号锁定后的标准恢复
当前版本基础恢复方式有两种：
1. 等待 `lock_until` 自然过期
2. 用户通过“找回密码”成功重置密码后，系统自动清空：
   - `login_fail_count`
   - `last_login_failure_at`
   - `lock_until`

### 5.2 手工核验 SQL
```sql
SELECT username, login_fail_count, last_login_failure_at, lock_until
FROM sys_user
WHERE username = '20260002';
```

## 六、验证方式
### 6.1 锁定验证
1. 对同一账号连续输入错误密码 5 次。
2. 使用正确密码再次登录，应返回“临时锁定”提示。
3. 管理员消息中心应出现 `SECURITY_ALERT`。

### 6.2 解锁验证
1. 对被锁账号执行找回密码。
2. 使用新密码重新登录，应成功。

### 6.3 限流验证
1. 使用同一 IP 连续发起 12 次以上登录请求。
2. 后续请求应返回“当前 IP 登录过于频繁，请稍后再试”。

## 七、当前边界
1. 当前告警通道仅有站内消息，尚未接入短信、邮件、企业微信、钉钉。
2. 当前没有自动封禁 IP 黑名单和图形验证码升级策略。
3. 当前没有统一观测平台和实时推送面板。

## 八、结论
当前仓库已经具备“风险日志 -> 阈值阻断 -> 站内告警 -> 管理排查 -> 找回密码自动解锁”的基础登录安全闭环。
