\# AGENTS.md



\## 非功能性硬要求

必须持续关注：

\- 权限边界

\- 数据隔离

\- 可扩展性

\- 可维护性

\- 审计与日志

\- 可测试性

\- 可部署性

\- 安全性



\## 实施边界

允许：

\- 自主创建目录、模块、脚本、文档

\- 自主抽取共享组件与公共能力

\- 自主补充合理但缺失的实现细节

\- 自主增加测试、校验、runbook、初始化脚本



不允许：

\- 未经论证地更换技术栈

\- 为了省事把敏感信息直接硬编码提交

\- 未验证就声称可运行

\- 只做空目录和空 README 就视为完成

\- 写死未来扩展空间

\- 引入与项目目标无关的框架和依赖



\## 文档同步规则

每完成一个里程碑，必须同步更新：

\- PLAN.md

\- Documentation.md

\- DECISIONS.md

\- CHANGELOG.md

\- 对应模块 README / runbook / SQL 说明



\## 验证规则

每一阶段至少执行：

\- 结构检查

\- 依赖安装检查

\- 后端构建/测试（如可行）

\- 前端构建/测试（如可行）

\- 数据库脚本检查

\- 路由/权限/配置一致性检查



\## 安全规则

\- 禁止把真实生产密码、token、私钥提交进仓库。

\- 本地开发配置应使用 .env.local、application-local.yml 或同类本地覆盖文件，并确保被 .gitignore 忽略。

\- 若必须在文档中展示配置，仅使用示例值。



\## 交付标准

项目只在以下条件满足时才可宣称“阶段完成”：

1\. 代码结构清晰

2\. 文档已同步

3\. 数据模型有依据

4\. 关键链路可解释

5\. 验证结果真实可追溯

6\. 未解决问题已写明

\## 本地数据库约束

当前本地开发数据库已经初始化完成。



AI 在开发过程中可以按需检查、修改、补充数据库结构与初始化脚本，但必须遵守以下规则：

\- 数据库类型：MySQL

\- host：127.0.0.1

\- port：3306

\- username：root

\- password：123456



执行要求：

\- AI 可以在本地开发环境中使用上述凭据连接数据库进行建库、建表、改表、初始化与验证。

\- 若需要调整数据库名、补充表结构、更新初始化脚本，必须同步更新文档与 SQL 脚本。

\- 不要把真实数据库凭据硬编码提交到生产配置中。

\- 优先通过 `.env.local`、`application-local.yml`、本地覆盖配置等方式使用该连接信息。

\- 若为本地开发方便临时写入配置，必须确保对应文件被 `.gitignore` 忽略，或在提交前改回示例值。



\## AI API Key 约束

项目后续可能接入 AI 能力，但当前阶段仅需预留接入能力，不直接写入真实 key。



要求：

\- 只预留配置项、接口适配层、调用边界与占位文档。

\- 不得提交真实 API Key。

\- 在 `.env.example` 或等价示例配置中保留类似以下字段：

&#x20; - `AI\_API\_BASE\_URL=`

&#x20; - `AI\_API\_KEY=`

&#x20; - `AI\_MODEL=`

\- 若项目中设计 AI 相关模块，必须通过环境变量读取配置。

## 文档语言与交付表达强制规则
本项目所有面向交付、验收、汇报、使用、维护的正式文档，必须以中文为主进行编写。

适用范围包括但不限于：
- 根目录文档
- docs/ 下全部正式文档
- 模块说明文档
- API 流程说明
- 数据库设计说明
- 测试文档
- 部署文档
- 运维文档
- runbook
- HANDOFF.md
- README.md
- 各模块 README

强制要求：
1. 文档标题、章节标题、正文说明、图表说明、步骤说明、风险说明、验收说明必须使用中文表达。
2. 可保留必要的英文技术名词、英文类名、英文接口名、英文表名、英文配置项，但必须配套中文解释。
3. 不允许出现“大段英文文档主体 + 少量中文点缀”的情况。
4. 不允许只把标题翻译成中文，正文仍然空泛或英文堆砌。
5. 文档必须详细、科学、可交付、可验收，不能写成口号、提纲、占位说明。
6. 文档必须站在甲方验收视角，写清楚：
   - 做了什么
   - 为什么这么做
   - 怎么实现的
   - 涉及哪些模块/数据/接口
   - 如何验证
   - 当前限制是什么
   - 后续如何扩展
## 强制续跑规则
当前项目目标不是“阶段性可交付”就停止，而是继续补齐当前明确需求，直到满足“真正完成”或“真实 blocker”条件。

AI 必须在每轮开始时执行：
1. 读取 PRD、ACCEPTANCE、EVALS、Documentation、HANDOFF
2. 形成缺口清单
3. 检查数据库是否真实存在、真实初始化、真实可用
4. 若数据库缺失或不完整，则自行创建、修复、导入脚本并验证
5. 再进入功能补齐与文档补齐

禁止：
- 跳过数据库检查
- 以“已有一期可用”作为停止理由
- 以“已有文档”作为停止理由
- 未做真实验证就宣称完成
## 文档质量强制规则
本项目文档质量直接关系交付成败，必须满足以下要求：

1. 详细
- 不能只有目录和几句概述
- 关键模块必须有背景、目标、流程、实现、数据、权限、验证、风险说明

2. 科学
- 表述要准确、结构化、可追溯
- 不能使用模糊措辞大量堆砌
- 不能用“基本支持、理论可行、后续可扩展”替代真实说明

3. 面向交付
- 文档要让甲方、老师、验收人、后续开发者都能看懂
- 要体现产品逻辑、工程逻辑、数据逻辑和验证逻辑

4. 与实现一致
- 文档必须与当前代码、数据库、接口、页面、脚本一致
- 不允许文档写了但系统没有
- 不允许系统有了但文档没写

5. 中文统一
- 发现已有英文文档时，必须逐步改写为中文正式文档
- 必要的英文术语保留，但必须给出中文语义说明

要求：

\- AI 在本地完成结构、代码、文档、脚本、数据库初始化与验证后，应将仓库 remote 指向该地址。

\- 若远端未配置，则添加为 `origin`。

\- 若远端配置错误，则修正并在 `Documentation.md` 与 `HANDOFF.md` 中记录。

\- 目标是让最终代码在远端仓库可见。

\- 若推送失败，必须写出精确失败命令、错误信息与后续人工 push 步骤。
